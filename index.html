Time for Something New

  • SMW is powerful and unique, has issues
  • Usability of Notion, Airtable, etc
  • SMW 20 years old, Wikibase almost 15
  • Many lessons learned

Many of you know the power of Semantic MediaWiki. It's a unique software that is highly flexible and can serve many use cases.

You also know it has some big issues. Despite all its promise, SMW never saw breakthrough success outside the MediaWiki world.

The competition hasn't stood still, and now highly popular tools like Notion and Airtable blow SMW out of the water when it comes to usability.

I've been maintaining SMW for over 15 years and created large parts of Wikibase. So I can say with some authority that we've learned many lessons along the way, and that it is time for something new.

NeoWiki

Open-source MediaWiki extension for structured data

  • 10x better usability than SMW
  • 10x better sustainability via modern architecture
  • 10% of features that deliver 90% of the value
  • Not aiming for feature parity

Weave in comparisons:

Schemas: In SMW you'd define this with a property page and hope people use the right type. Here the schema enforces it.

Views/queries: This is roughly what #ask does, except the query is Cypher and the result format is a first-class UI component.

JSON storage: Like Wikibase, data is stored as JSON in revision slots, not in a custom SQL schema.

Forms: Wikibase has forms but no schemas or constraints. SMW has PageForms as a separate bolt-on. Here it's native.

ECHOLOT

European Cultural Heritage: Optimized Linked Open Tools

EU-funded project for creation, reuse, and dissemination of high-quality, semantically rich cultural heritage data accessible to scholars and institutions of any size.

Built on knowledge graphs and open-source software (MediaWiki suite) for FAIR data principles and integration with Europeana, DS4CH, and Wikidata.

Funded by the European Union

NeoWiki vs Semantic MediaWiki

  • Native schemas with types, constraints, defaults
  • Native form-based editing, derived from schemas
  • Multiple subjects per page as a native concept
  • JSON storage in revision slots
  • Standard graph DB and query language
  • Modern codebase with full test coverage

NeoWiki vs Wikibase

  • Display UIs
  • Native schemas and constraints
  • Forms
  • Property graph model
  • QLever instead of Blazegraph
  • "Inline queries"

NeoWiki vs Bucket

Bucket

  • Similar philosophy: serve common cases, keep code lean
  • Designed for the gaming wiki use cases
  • Optimized for power users who write Lua and SQL

NeoWiki

  • High-UX views and forms
  • Schemas with validation
  • Graph database and queries
